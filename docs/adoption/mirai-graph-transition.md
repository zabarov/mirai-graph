# GrowGraph To Mirai Graph Transition

Status: alpha.9 transition note

Mirai Graph was formerly developed under the working name GrowGraph.

The current naming model is:

- `Mirai` is the broader AI direction.
- `Mirai Graph` is the public standard, model and methodology.
- `mirai-graph` is the repository, package and primary CLI slug.
- `GrowGraph` is a historical name and temporary compatibility alias.
- `EOG` remains a scientific continuity term for earlier research framing.

The strategic goal of the rename is not cosmetic. It separates the future-facing
Mirai direction from the former GrowGraph working name and positions Mirai Graph
as the public graph standard for governed AI work: AI employees, projects,
organizations and research systems coordinated through state, evidence, gates,
feedback and controlled evolution.

## What Changed

- Package name: `mirai-graph`.
- Primary CLI: `node packages/cli/mirai-graph.js`.
- Preferred package manifest: `mirai-graph-package.json`.
- Public docs and examples use Mirai Graph naming.

## What Still Works

During the alpha transition, these legacy paths still work:

- `node packages/cli/growgraph.js`
- `node packages/cli/validate-growgraph.js`
- `growgraph-package.json`

They are compatibility surfaces, not the preferred public interface.

## Migration Steps

1. Rename local package manifests from `growgraph-package.json` to
   `mirai-graph-package.json`.
2. Replace direct validator calls with
   `node packages/cli/validate-mirai-graph.js`.
3. Replace unified CLI calls with `node packages/cli/mirai-graph.js`.
4. Keep old generated reports as historical evidence; do not rewrite evidence
   payloads unless they are regenerated by the current tooling.

## Schema Namespace

Existing schema `$id` values may still use the previous namespace during the
transition. Mirai Graph will move schema namespaces only after the public domain
decision is confirmed. Schema `$id` changes should be treated as a compatibility
event, not as a cosmetic text edit.

## Boundary

The rename does not change the standard semantics:

- generated context is not canonical state;
- evidence and proposals do not authorize canonical updates;
- feedback does not automatically change behavior;
- readiness is not execution;
- compatibility aliases are not new standards.
